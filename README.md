# claude-setup

Базовый шаблон для настройки AI-окружения проекта. Копируешь в корень репозитория — и сразу получаешь рабочий pipeline: PM-оркестрация, аналитика через Spec-Kitty, управление задачами через Backlog.md, сессии через Entire, worktree-менеджмент через Superset.

---

## Что внутри

```
.claude/
├── CLAUDE.md              # Системный промпт PM-агента
├── settings.json          # Хуки Claude Code (Entire интеграция)
├── agents/
│   ├── analyst.md         # SA — системный аналитик (Spec-Kitty цикл)
│   ├── developer.md       # Dev — разработчик (задача → PR)
│   ├── qa.md              # QA — тестировщик (PASS/FAIL с evidence)
│   └── git-sync.md        # Git Sync — синхронизация worktree
└── commands/
    └── spec-kitty.*.md    # 14 команд Spec-Kitty (specify, plan, checklist, tasks, ...)

.backlog/
└── config.yml             # Конфиг Backlog.md MCP

.entire/
├── settings.json          # Конфиг Entire (checkpoints, сессии)
└── .gitignore             # Игнор runtime-данных Entire

.superset/
└── config.json            # Конфиг Superset (worktree менеджер)

.gitignore                 # Общий gitignore (runtime-данные исключены, конфиги отслеживаются)
```

## Быстрый старт (ШАГ 0 — Сетап проекта)

### 1. Склонировать шаблон в свой проект

```bash
# Вариант A — скопировать содержимое в существующий проект
git clone git@github.com:ai-oxudevelopment/claude-setup.git /tmp/claude-setup
cp -r /tmp/claude-setup/.claude /tmp/claude-setup/.backlog \
      /tmp/claude-setup/.entire /tmp/claude-setup/.superset \
      /tmp/claude-setup/.gitignore \
      /path/to/your/project/
rm -rf /tmp/claude-setup

# Вариант B — использовать как основу нового проекта
git clone git@github.com:ai-oxudevelopment/claude-setup.git my-new-project
cd my-new-project
git remote set-url origin git@github.com:your-org/your-repo.git
```

### 2. Установить зависимости

```bash
# Backlog.md — управление задачами через MCP
npm install -g backlog.md

# Spec-Kitty — генерация спецификаций (skill для Claude Code)
npm install -g spec-kitty

# Entire — сессии и checkpoints
npm install -g @anthropic/entire

# Superset — worktree менеджмент (опционально)
# Установка через https://superset.dev
```

### 3. Подготовить git

```bash
# Скачайте все репозитории над которыми планируете работать
git clone ...
git submodule update --init --recursive

# Убедитесь что проект запускается сам по себе — это сильно сэкономит время отладки
npm run dev  # или ваша команда запуска
```

### 4. Настроить MCP

Создай `.claude/mcp.json` (или `~/.claude/mcp.json` для глобального конфига):

```json
{
  "mcpServers": {
    "backlog": {
      "command": "backlog",
      "args": ["mcp", "start"],
      "env": {
        "BACKLOG_CWD": "/absolute/path/to/your/project"
      }
    }
  }
}
```

### 5. Проверить что всё работает

```bash
# Обновить имя проекта в Backlog
sed -i '' 's/my-project/your-project-name/' .backlog/config.yml

# Запустить Claude Code
claude

# Внутри Claude Code — проверить MCP-серверы
/mcp
```

Через команду `/mcp` убедитесь что все MCP-серверы доступны. Если что-то не подключилось — проверьте пути в `.claude/mcp.json`.

## Бизнес-процесс: пошаговая инструкция

### ШАГ 1 — Копирование промпта

Сразу после запуска Claude Code — **вставьте в чат содержимое `CLAUDE.md`**. По каким-то причинам Claude Code не всегда следует инструкциям из файла автоматически, и нужно явно записать это в контекст в самом начале сессии.

После этого PM-агент автоматически:
1. Запустит **Git Sync агента** — синхронизация всех проектов, подтягивание `.claude/` из main-папки, git pull по субмодулям
2. Проверит подключение к **Backlog MCP**
3. Создаст задачу `[SYNC]` и зафиксирует результат

### ШАГ 2 — Описать задачу

Потратьте время и опишите задачу с позиции результата и как вы его будете тестировать. Используйте шаблон:

```markdown
**Для кого**
[кто будет пользоваться результатом]

**Что делаем**
[суть задачи в 1-2 предложениях]

**Как сейчас работает**
[текущее поведение системы]

**Какие есть сложности**
[боли, проблемы, ограничения]

**Как должно быть**
[целевое поведение, конкретно]

**Как буду проверять**
[шаги тестирования: открыть X → нажать Y → увидеть Z]

**Полезные документы**
[ссылки на доки, схемы, API]

**Дополнительные сведения**
[стек, ограничения, зависимости]
```

> В идеале — сгенерировать на старте хорошую и понятную фичу с импактом. Из практики — учитывайте зону влияния AI-агента и его возможности.

### ШАГ 3 — Ответить на вопросы PM

PM задаст 6 вопросов в рамках INTAKE. **Качество ответов на этом этапе — это 90% успеха генерации.**

Советы:
- Давайте максимально подробный и развёрнутый ответ, всегда с привязкой к результату
- Можете скопировать вопрос и подумать над ответом через другого AI-агента (Perplexity, Claude.ai)
- Обязательно опишите **критерии приёмки**: как будете проверять работу, на что обращать внимание, где смотреть, как тестировать

### ШАГ 4 — Автономный SA-аналитик

PM по протоколу заводит задачу через MCP и запускает SA-аналитика. На фоне стартует автономный AI-агент, который через Spec-Kitty генерирует SDD-спецификацию.

**Мониторинг прогресса:**

```bash
# Просмотр dashboard Spec-Kitty (прогресс генерации спецификации)
spec-kitty dashboard

# Просмотр бэклога задач
backlog board

# Веб-интерфейс со всеми артефактами
backlog browser
```

Spec-Kitty Dashboard покажет прогресс по 4 разделам:
- **Specification** — спецификация фичи
- **Plan** — технический план реализации
- **Checklist** — чек-лист приёмки
- **Tasks** — декомпозиция на задачи

Когда все 4 раздела заполнены — SA завершил работу. Через `backlog board` или `backlog browser` можно просмотреть план задач.

### ШАГ 4.bug — Если что-то пошло не так

Entire предоставляет полный лог контекста и систему чекпоинтов. Можно попросить AI-агента откатиться к любой точке процесса.

```bash
# Просмотр лога чекпоинтов
entire log

# Откат к конкретному чекпоинту
entire rewind <checkpoint-label>
```

**Система чекпоинтов по умолчанию:**

| Чекпоинт | Момент |
|----------|--------|
| `sa-start-{TASK_ID}` | SA начал работу |
| `sa-specify-{TASK_ID}` | Specification сгенерирована |
| `sa-plan-{TASK_ID}` | Plan сгенерирован |
| `sa-checklist-{TASK_ID}` | Checklist сгенерирован |
| `sa-complete-{TASK_ID}` | Все этапы SA завершены |
| `dev-start-{TASK_ID}` | Dev начал реализацию |
| `dev-plan-{TASK_ID}` | Dev написал план |
| `dev-pr-{TASK_ID}` | PR открыт |
| `qa-start-{TASK_ID}` | QA начал тестирование |
| `qa-complete-{TASK_ID}` | QA вынес вердикт PASS |
| `qa-fail-{TASK_ID}` | QA вынес вердикт FAIL |

### ШАГ 5 — Ревью сгенерированных артефактов

Важный этап после работы SA-аналитика. Все артефакты доступны в браузере:

```bash
backlog browser
```

В веб-интерфейсе вы увидите:
- **Documents** — спецификация, план, решения
- **Tasks** — декомпозированные задачи с критериями приёмки
- **Decisions** — принятые архитектурные решения

Проверьте что задачи адекватно декомпозированы и критерии приёмки соответствуют вашим ожиданиям. После подтверждения — PM запустит разработку.

## Архитектура агентов

```
Человек
  │
  ▼
┌──────────┐     ┌──────────────┐
│    PM    │────▶│  Backlog MCP │
│ (CLAUDE) │     └──────────────┘
└────┬─────┘
     │ Task()
     ├────────────────────────────────┐
     ▼                                ▼
┌──────────┐  Spec-Kitty         ┌──────────┐
│    SA    │─────────────────────│  SCRUM   │
│ Analyst  │  specify → plan →   │  Master  │
└──────────┘  checklist → tasks  └────┬─────┘
                                      │ CHECKPOINT (ждёт "да")
                                      ▼
                                ┌──────────┐
                                │   Dev    │──▶ PR
                                └────┬─────┘
                                     ▼
                                ┌──────────┐
                                │    QA    │──▶ PASS / FAIL
                                └──────────┘
```

### Роли агентов

| Агент | Файл | Что делает | Что НЕ делает |
|-------|------|-----------|--------------|
| **PM** | `CLAUDE.md` | Оркестрирует процесс, управляет Backlog через MCP, вызывает агентов | Не пишет код, не анализирует, не читает кодовую базу |
| **SA** | `agents/analyst.md` | Генерирует спецификации через Spec-Kitty, создаёт подзадачи | Не создаёт файлы вручную, не пишет код |
| **Dev** | `agents/developer.md` | Реализует одну задачу, открывает PR | Не берёт больше одной задачи, не закрывает без PR |
| **QA** | `agents/qa.md` | Тестирует по критериям, выносит PASS/FAIL | Не исправляет баги, вердикт только с evidence |
| **Git Sync** | `agents/git-sync.md` | Синхронизирует worktree, git pull, субмодули | Не удаляет entire-ветки |
| **SCRUM** | — | Верифицирует бэклог, Token Budget Gate (175k токенов/задача) | Не переписывает задачи с нуля |

### Pipeline фазы

| Фаза | Агент | Вход | Выход |
|------|-------|------|-------|
| INTAKE | PM | Описание задачи | 6 ответов на вопросы |
| Фаза 1 | SA (Analyst) | INTAKE + Spec-Kitty | Спецификация, план, чеклист, подзадачи |
| Фаза 2 | SCRUM Master | Подзадачи | Верифицированный бэклог |
| CHECKPOINT | PM | SCRUM-REPORT | Явное "да" от человека |
| Фаза 3 | Developer | Задача из бэклога | PR в main |
| Фаза 4 | QA | PR + критерии | PASS/FAIL с evidence |
| Фаза 5 | PM | QA-REPORT | QA Gate (done / доработка) |

## FAQ

### Зачем нужен PM-агент?

Бизнес-процесс разработки — это 90% кодогенерация. Но очень много времени и фокуса уходит на постоянное двигание задачи по статусам — это рутина. PM-агент берёт её на себя: получает фичу и доводит до ума, управляя процессом через MCP. Он создаёт задачи, заполняет описания, вызывает агентов (SA, Dev, QA) и контролирует исполнение.

### Почему PM пишет что "не может" сделать задачу?

PM никогда сам не исполняет задачу — у него ограничение. Он всегда создаёт задачу в Backlog, заполняет описание и вызывает нужного агента. PM только контролирует.

### Чем PM отличается от Dev-агента?

У PM полностью отключена возможность вести разработку. Он умеет только управлять MCP-сервером Backlog: прочитать задачу, добавить описание, разделить на подзадачи, вызвать агента, оставить комментарий. Его единственный инструмент — `Task()` для вызова других агентов.

### Если PM-агент уходит в исполнение кода?

К сожалению такое бывает — Claude Code заточен на самостоятельное написание кода. Единственный способ — остановить генерацию и напрямую поставить ограничения в чат. Промпт `CLAUDE.md` постоянно дорабатывается чтобы минимизировать такое поведение.

### Какой уровень декомпозиции задач?

AI-агенты могут генерировать код без ограничений по времени. Главное условие — предсказать контекст разработки. Если модель обрабатывает 200k токенов, нужно декомпозировать так, чтобы одна задача заняла не больше **175k токенов**. SCRUM-агент проверяет это через Token Budget Gate.

### Как управляется контекст PM-агента?

Как и в жизни, у PM очень много простых изменений статуса, которые быстро заполняют контекст. Эту задачу решает [Entire](https://entire.dev) — управляемый контекст AI-модели в формате Git-истории. Checkpoints позволяют откатываться к любому моменту процесса.

### Что лучше использовать для работы с PM-агентами?

Удобно работать с [Superset](https://superset.dev) и Codex — их интерфейсы заточены именно для работы с AI-агентами, а не с кодом. Это ощущается в подходе как выстраиваешь работу и на что обращаешь внимание.

## Кастомизация

### Добавить своего агента

Создай файл `.claude/agents/my-agent.md` по шаблону:

```markdown
# MY AGENT -- АВТОНОМНЫЙ АГЕНТ

## ИДЕНТИЧНОСТЬ
Ты -- автономный агент. Одна задача, один результат.

## СТАРТОВЫЙ ПРОТОКОЛ
1. backlog__task_get(TASK_ID)
2. entire checkpoint "my-agent-start-{TASK_ID}"
3. ... твоя логика ...

## ФИНАЛЬНЫЙ ОТЧЁТ
backlog__task_update(TASK_ID, notes="[MY-AGENT-REPORT] ...")
```

### Изменить CLAUDE.md

`CLAUDE.md` — системный промпт PM-агента. Можно изменить:
- Фазы pipeline
- Вопросы INTAKE
- Правила верификации
- Checkpoint-стратегию

### Промпты агентов

Промпты постоянно дорабатываются. Актуальные версии:

| Агент | Файл | Описание |
|-------|------|----------|
| PM | `.claude/CLAUDE.md` | Оркестрация, делегирование, контроль |
| SA | `.claude/agents/analyst.md` | Аналитика через Spec-Kitty |
| Git Sync | `.claude/agents/git-sync.md` | Синхронизация кода и worktree |
| Dev | `.claude/agents/developer.md` | Реализация задачи → PR |
| QA | `.claude/agents/qa.md` | Тестирование, PASS/FAIL вердикт |
